## 1. anon key 放前端 —— 這是「正常設計」,不是低級錯誤

關鍵要先澄清一個常見誤解:**Supabase 的 anon key 本來就是設計成公開放在前端的**,跟你網站放 Google Maps API key 在前端一樣。它不是密鑰,它是「我是哪個專案」的識別證。Supabase 官方文件也明說可以公開。

所以「能逆向出它用 Supabase」跟「anon key 外洩」其實是兩件事:

- **能看出是 Supabase**:這跟 key 無關。任何前端框架(Vite/React)打包後,JS 裡都看得到 `supabaseClient` 的 import、`.supabase.co` 的 URL、`rest/v1`、`auth/v1` 這些路徑特徵。就算他用付費版、自架,只要呼叫是從瀏覽器發出的,DevTools 的 Network 分頁一打開就現形了。**任何 client-side 直連資料庫的架構,API 結構天生對使用者透明。**
- **真正的問題在 RLS(Row-Level Security)**:anon key 公開沒事,前提是你在資料庫層設好權限。Supabase 的安全模型是「key 公開 + RLS 把關」。我能撈走全部 1749 筆,代表 `startups` 這張表的 RLS 要嘛沒開,要嘛 policy 設成 `SELECT` 對 anon 全開。
